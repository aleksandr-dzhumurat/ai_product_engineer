## LogLoss

Log Loss это синоним Binary Cross-Entropy (BCE).

Штрафует ли **logloss** корректно предсказанные метки классов? И если да, то зачем?

$$L(y, F(x)) = -y \log(p) - (1-y)\log(1-p)$$

где $p = \frac{1}{1 + e^{-F(x)}}$

Антиградиент: $r_{im} = y_i - p_i$

Для предсказания классов нужен порог (**threshold**), ошибки классификации (precision, recall) - не гладкие.

Модель выдает не метку, а вероятность - при пороге 0.5 правильный прогноз в 0.51 может **наказываться за слабую уверенность**. При $y_{true} = 1$ кривая $−log(p)$ выпуклая и крутая вблизи p = 0,51 — модель сильно наказывается за "неуверенность" в приваильном ответе. По мере того, как p → 1, потери плавно приближаются к 0.

# Логистическая регрессия: простой кейс

**Данные:** класс 0 ($x < 4$ кг), класс 1 ($x > 4$ кг)

**Модель:**

$$P(y = 1 | x) = \sigma(wx + b) = \frac{1}{1 + e^{-(wx+b)}}$$

**Параметры:** $w$ и $b$

**Граница разделения:** сигмоид = 0.5 при $wx + b = 0$

$$x = -\frac{b}{w} = 4 \quad \Rightarrow \quad b = -4w$$

для $x < 4 \Rightarrow y = 0$ нужно $w > 0$

**Ответ:**

$$b = -4w, \quad w > 0$$

Например: $w = 1, b = -4$ или $w = 10, b = -40$


# Classification metrics

Хороший разбор [в статье про метрики](https://habr.com/ru/company/ods/blog/328372/)

Confusion matrix (матрица ошибок):

|  | **Predicted Positive** | **Predicted Negative** |
| --- | --- | --- |
| **Actual Positive (1s)** | TP | FN |
| **Actual Negative (0s)** | FP | TN |
* FP → Type I error.
* FN → Type II error.

В машинном обучении F1-мера (F1 score) — это среднее гармоническое точности (precision) и полноты (recall).

$$F1 = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

Точность (Precision): Эта метрика измеряет, какая доля объектов, которые модель **ошибочно** классифицировала как положительные.

$$\text{Precision} = \frac{TP}{TP + FP}$$

Полнота (Recall): Эта метрика измеряет, какую долю объектов из всех реально существующих положительных примеров модель смогла найти и правильно определить.

$$\text{Recall} = \frac{TP}{TP + FN}$$

✅ Высокая полнота (High recall) -> модель успешно находит и «отлавливает» почти все положительные примеры, доля False Negatives мала. Recall используют когда цена False Negative намного выше, чем цена «ложной тревоги», то есть модель отлично находит "позитивы"

- **Медицинская диагностика** (рак, COVID-19 и др.) — пропустить больного пациента недопустимо.
- **Fraud Detection** — лучше лишний раз сработать на подозрительную транзакцию, чем упустить мошенничество.
- **Системы безопасности (Intrusion Detection)** — ложная тревога предпочтительнее пропущенной угрозы.

*Note:* Recall, иногда называют **Sensitivity** или **True Positive Rate (TPR)** —  Используют при построении **ROC кривой**


**Trade-off:** Высокий **recall** может увеличить количество **false positives** (будет низкий **precision**).

Для измерения способности отлавливать "негативы" вычисляют **Specificity (TNR)**.

$$TNR = \frac{TN}{TN + FP}$$

Высокая специфичность → Хорошо справляется с исключением негативов (например, обнаружение спама - чтобы важные письма не попадали в спам).

**Accuracy**: обобщает оба класса ошибок.

$$\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}$$

⚠️ Может вводить в заблуждение на **несбалансированных наборах данных**. Модель, предсказывающая все отрицательные значения на наборе данных, состоящем на 95% из отрицательных значений, достигает 95% точности, но не обнаруживает ни одного положительного значения.

## F1 Score

$$F1 = 2 \times \frac{(TP/(TP + FP)) \times (TP/(TP + FN))}{(TP/(TP + FP)) + (TP/(TP + FN))}$$

⚠️  Реализации метрик есть в sklearn.

```python
from sklearn.metrics import balanced_accuracy_score, f1_score, recall_score

balanced_acc = balanced_accuracy_score(y_true, y_pred)

f1 = f1_score(y_true, y_pred)
```

Use F1 when classes are imbalanced (e.g., fraud detection, medical diagnosis).

- Если у false positives и false negatives **разная стоимость ошибки**, вместо F1 нужно выбрать **precision** или **recall**:
    - **High Precision?** → модель минимизирует **false positives**.
    - **High Recall?** → модель минимизирует  **false negatives**.

## ROC Curve (ROC-AUC)

ROC-AUC (Receiver Operating Characteristic - Area Under Curve) — это метрика для оценки качества бинарных классификаторов, особенно полезная для моделей, которые выдают не просто класс (0 или 1), а вероятность принадлежности к классу.

## Интерпретация ROC-AUC

- **AUC = 1.0** — идеальный классификатор (все положительные имеют score выше, чем все отрицательные)
- **AUC = 0.9-1.0** — отличная модель
- **AUC = 0.8-0.7** — хорошая модель
- **AUC = 0.5** — модель не лучше случайного угадывания
- **AUC < 0.5** — модель хуже случайного угадывания (возможно, классы перепутаны)

В отличие от Precision, Recall и F1, которые зависят от выбранного порога (threshold) для классификации, ROC-AUC оценивает качество модели независимо от порога. Это позволяет понять, насколько хорошо модель различает классы в целом (то есть по всем возможным порогам сразу).

Когда это важно: рекомендательные системы. Не важно, насколько хорошо модель различает товары с вероятностью 0.01 и 0.02, важно отранжировать товары по скору. Важна Precision@10 — из 10 показанных товаров сколько реально купят

Когда использовать

- False positives (FP) и false negatives (FN) имеют **равную стоимость ошибки** или **FP менее важны**, т.к. FPR нормализуется по общему количеству негативов в выборке.
- **balanced classes**
- рекоммендательные системы - лучший кейс. Важно чтобы лучшие товары были первыми в выдаче.

Когда не использовать

ROC AUC не подходит, когда **не важно** насколько хорошо модель различает классы в целом, важен перформанс на конкретных порогах - это задачи медицины, фильтрации спама. Или фрод: Нам важно найти иголку в стоге сена, а не способность различать легитимные транзакции между собой:
* Сильный дисбаланс классов (один класс редкий)
* Асимметричная цена ошибок (FP и FN имеют разную стоимость)
* Важен только топ (top-K рекомендаций/результатов)

ROC-AUC **может выглядеть высокой** на несбалансированных данных, потому что она фокусируется на ранжировании и false positive rate, которые определяются мажоритарным классом (обычно это негативы `0`), часто скрывая плохую производительность на миноритарном классе (который мы и предсказываем).

Доля ложноположительных ответов (FPR) остается низкой, даже если многие отрицательные объекты классифицированы неверно (FP), так как знаменатель — общее количество отрицательных объектов — огромен.

**2. Хороший ранжир ≠ хорошие предсказания**
ROC-AUC измеряет, насколько хорошо модель ранжирует положительные примеры выше отрицательных, но не то, насколько точно она предсказывает положительный класс при используемом пороге. Модель может иметь высокий ROC-AUC, но при этом показывать очень плохие precision или recall для миноритарного класса.

На практике важна производительность при конкретном пороге принятия решения (например, частота срабатывания алертов, ограничения по затратам).

Что такое порог? Модель выдаёт вероятность, например $0.73$. Ты сам выбираешь порог: «если вероятность > 0.5, это позитивный класс». В зависимости от задачи этот порог может быть совсем другим — 0.1 или 0.9.

⚠️ ROC-AUC усредняет производительность по всем порогам, многие из которых неактуальны в условиях сильного дисбаланса.

ROC-AUC вычисляет качество модели при каждом возможном пороге от 0 до 1, и усредняет. Грубо говоря — «насколько хороша модель в целом, вне зависимости от порога».
В чём проблема при сильном дисбалансе?
Допустим, позитивных примеров 1%, негативных 99%. Мы ранжируем объекты по скору и начинаем перебирать пороги начиная с самых низких скоров. Большинство порогов, которые ROC-AUC учитывает, дают тебе режим «почти всё — негатив». В реальности тебя интересует только узкий диапазон порогов, где модель начинает находить редкий позитивный класс. ROC-AUC «разбавляет» этот важный диапазон всеми остальными, неактуальными.

Конкретный пример:

* Задача: детекция мошенничества (fraud), 0.1% позитивных
* Тебе важно: при пороге, который даёт 100 алертов в день, сколько из них реальное мошенничество?

ROC-AUC на это не отвечает — он усредняет по всем порогам, включая те, где алертов 0 или 100 000

### Главный вывод

При сильном дисбалансе ROC-AUC нечувствителен к абсолютному числу TN, потому что FPR нормирует FP на общее число негативов. Проблема в том, что при большом числе негативов даже значительное абсолютное число FP даёт маленький FPR, и это не отражается в ROC-кривой.

Кривая ROC строится по осям TPR, FPR

По вертикали TPR (True Positive Rate) — Чувствительность
```
TPR = TP / (TP + FN)
```
Это доля правильно предсказанных положительных примеров среди всех положительных. По сути, это тот же **Recall**.

По горизонтали FPR (False Positive Rate)
```
FPR = FP / (FP + TN)
```
Это доля неправильно предсказанных положительных примеров среди всех отрицательных.

$$FPR = \frac{FP}{FP + TN}$$

TN очень много → FPR остаётся маленьким даже при нескольких FP →
кривая выглядит красиво, хотя модель на практике почти бесполезна.

Именно поэтому при дисбалансе смотрят на **PR-AUC** (Precision-Recall AUC) —
она не использует TN вообще и честно показывает качество на миноритарном классе.

Как это работает: **P-R Curve** строит график в осях Precision vs. Recall, фокусируясь на **positive class performance**. Хорошо работает для 

- **imbalanced datasets** (например, fraud detection), позитивный класс редкий.
- когда высока цена **false positives** (положили в спам письмо от начальника - очень плохо)
- важен **precision and recall trade-offs**.

| **Scenario** | **ROC Curve** | **P-R Curve** |
| --- | --- | --- |
| Balanced dataset (50-50) | ✅ Yes | ❌ No |
| Imbalanced dataset (e.g., 1% positives) | ❌ No | ✅ Yes |
| Medical diagnosis (minimizing false negatives) | ✅ Yes | ✅ Yes |
| Fraud detection (rare class) | ❌ No | ✅ Yes |
| When FP rate is misleading | ❌ No | ✅ Yes |


**Вопрос**: Если возвести предсказания алгоритма в квадрат, что станет с метрикой ROC AUC?

**Ответ**: Для вероятностей в диапазоне [0,1] возведение в квадрат просто сжимает их к 0, но монотонность сохраняет, поэтому порядок не меняется -> ROC AUC останется неизменным, так как метрика оценивает правильность ранжирования алгоритма - которое не изменится при возведении в квадрат. 


# Кейс: "lazy" model

"Ленивая" модель предсказывает только мажоритарный класс (всегда 1 или всегда 0).

**Проблема**: как обнаружить "ленивую" модель? Диагностика классификации: тест инвертирования классов

Основные метрики

$$\text{Precision} = \frac{TP}{TP + FP} \quad \text{Recall} = \frac{TP}{TP + FN}$$

$$\text{F1} = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

Пример несбалансированного датасета

- Класс 1: 30,000 примеров (83.3%)
- Класс 0: 6,000 примеров (16.7%)
- **Итого:** 36,000 примеров

---

###  Сценарий 1: "Ленивая" модель

Поведение: модель всегда предсказывает класс 1

| | Pred 0 | Pred 1 |
|---------|--------|--------|
| **True 0** | 0 | 6,000 |
| **True 1** | 0 | 30,000 |

Метрики:
- TP = 30,000, FP = 6,000, FN = 0, TN = 0
- Recall = $\frac{30,000}{30,000 + 0} = 1.0$ ✓
- Precision = $\frac{30,000}{30,000 + 6,000} = 0.833$ ✓

⚠️ Метрики выглядят хорошо, но модель бесполезна!

### Тест: инвертирование классов (0 ↔ 1)

После инверсии модель всё ещё предсказывает "старый класс 1" = **новый класс 0**

| | Pred 0 (новый) | Pred 1 (новый) |
|---------|--------|--------|
| **True 0** | 30,000 | 0 |
| **True 1** | 6,000 | 0 |

Новые метрики:
- TP = 0, FP = 0, FN = 6,000
- Recall = $\frac{0}{0 + 6,000} = 0$ ❌
- Precision = undefined (0/0) ❌

На практике: проверяй confusion matrix
- Строка или столбец полностью нулевые
- Диагональ имеет один нулевой элемент

Реализация на python

```python
from sklearn.metrics import confusion_matrix
import seaborn as sns

cm = confusion_matrix(y_true, y_pred)
sns.heatmap(cm, annot=True, fmt='d')
```

**Вывод:** Метрики упали в ноль → модель ленивая!

Почему так происходит? При инвертировании классов

$$\begin{aligned}
TP_{new} &= TN_{old} \\
FP_{new} &= FN_{old} \\
FN_{new} &= FP_{old} \\
TN_{new} &= TP_{old}
\end{aligned}$$

Для ленивой модели: $TN_{old} = 0$ → $TP_{new} = 0$ → метрики падают в 0

## Сценарий 2: Хорошо обученная модель

Исходная confusion matrix

| | Pred 0 | Pred 1 |
|---------|--------|--------|
| **True 0** | 5,500 | 500 |
| **True 1** | 600 | 29,400 |

Метрики:
- Recall = $\frac{29,400}{29,400 + 600} = 0.98$
- Precision = $\frac{29,400}{29,400 + 500} = 0.983$

После инвертирования классов

| | Pred 0 | Pred 1 |
|---------|--------|--------|
| **True 0** | 29,400 | 600 |
| **True 1** | 500 | 5,500 |

Новые метрики:
- Recall = $\frac{5,500}{5,500 + 500} = 0.917$
- Precision = $\frac{5,500}{5,500 + 600} = 0.901$

**Вывод:** Метрики остаются высокими → модель действительно обучена!

Реализация инвертирования классов на python

```python
y_true_inv = 1 - y_true
y_pred_inv = 1 - y_pred
recall_inv = recall_score(y_true_inv, y_pred_inv)
if recall_inv < 0.1:
    print("⚠️ Модель предсказывает только один класс!")
```


| Признак | Ленивая модель | Обученная модель |
|---------|----------------|------------------|
| **Recall на мажоритарном классе** | ≈ 1.0 | < 1.0 (есть ошибки) |
| **Precision** | ≈ доля мажоритарного класса | > доля класса |
| **После инвертирования** | Метрики → 0 | Метрики остаются высокими |
| **Confusion matrix** | Одна строка/столбец = 0 | Все ячейки заполнены |

---

Диагностическая таблица

| Метрики | Интерпретация | Действие |
|---------|---------------|----------|
| Recall=1.0, Precision=доля_класса | 🚨 Ленивая модель | Проверить CM, изменить веса |
| Recall≈1.0, Precision>>доля_класса | ✅ Хорошая модель на мажоритарном | Проверить минор. класс |
| Recall<0.9, Precision>0.9 | ⚠️ Консервативная модель | Порог вероятности? |
| Обе метрики ~0.5 | 🤷 Рандомная модель | Переобучить с нуля |

⚠️  Почему Precision = доля мажоритарного класса?

Если модель всегда предсказывает класс 1:

$$\text{Precision} = \frac{TP}{TP + FP} = \frac{n_1}{n_1 + n_0} = \frac{n_1}{n_{total}}$$

где $n_1$ - количество примеров класса 1

**Пример: кейс с дисбалансом**

Данные: 45K класса 1, 5K класса 0  
Метрики: Precision = 90%, Recall = 100%

Recall = 100% → TP = 45,000

Тут как раз видим долю мажоритерного класса.

$$\text{Precision} = \frac{45000}{45000 + FP} = 0.9 \quad \Rightarrow \quad FP = 5000$$

Confusion Matrix:
|  | Pred 0 | Pred 1 |
|--|--------|--------|
| Actual 0 | 0 | 5000 |
| Actual 1 | 0 | 45000 |

**Гипотеза:** модель предсказывает ВСЁ как класс 1. Высокие метрики — следствие дисбаланса. Деплоить такое нельзя - надо проверить что модель не различает классы (например через инвертирование классов).


## Строим ROC-AUC на примере


Пошаговый процесс построения ROC-AUC

### Шаг 1: Собираем предсказания модели

Модель выдала вероятности (scores) для каждого объекта:

| Id | Score | Actual Class |
|----|-------|--------------|
| 1  | 0.9   | 0            |
| 2  | 0.8   | 1            |
| 3  | 0.2   | 1            |
| 4  | 0.6   | 0            |
| 5  | 0.1   | 1            |
| 6  | 0.3   | 0            |
| 7  | 0.5   | 1            |
| 8  | 0.0   | 0            |

### Шаг 2: Сортируем по убыванию score

| Id | Score | Actual Class |
|----|-------|--------------|
| 1  | 0.9   | 0            |
| 2  | 0.8   | 1            |
| 4  | 0.6   | 0            |
| 7  | 0.5   | 1            |
| 6  | 0.3   | 0            |
| 3  | 0.2   | 1            |
| 5  | 0.1   | 1            |
| 8  | 0.0   | 0            |

### Шаг 3: Строим кривую

Начинаем с точки (0, 0) и идем по отсортированным примерам:

- **Если встретили класс 1** → делаем шаг **ВВЕРХ** (вертикальная линия)
  - Увеличивается TPR (нашли еще один положительный пример)
  
- **Если встретили класс 0** → делаем шаг **ВПРАВО** (горизонтальная линия)
  - Увеличивается FPR (ошибочно классифицировали отрицательный как положительный)

**Количество шагов:**
- **m** вертикальных линий = число примеров класса 1
- **n** горизонтальных линий = число примеров класса 0

### Шаг 4: Вычисляем площадь

**AUC** (Area Under Curve) — это площадь под построенной кривой. Она всегда находится в диапазоне от 0 до 1.

One more example

Say you have 5 examples with predicted scores and true labels:

| Example | Score | Label |
|---------|-------|-------|
| A | 0.9 | + |
| B | 0.7 | + |
| C | 0.6 | − |
| D | 0.4 | + |
| E | 0.2 | − |

**Step 1 — sweep thresholds from high to low**

At each threshold, everything above it is predicted positive.

| Threshold | Predicted + | TP | FP | TPR | FPR |
|-----------|-------------|----|----|-----|-----|
| > 0.9 | none | 0 | 0 | 0/3 = 0.00 | 0/2 = 0.00 |
| > 0.7 | A | 1 | 0 | 1/3 = 0.33 | 0/2 = 0.00 |
| > 0.6 | A, B | 2 | 0 | 2/3 = 0.67 | 0/2 = 0.00 |
| > 0.4 | A, B, C | 2 | 1 | 2/3 = 0.67 | 1/2 = 0.50 |
| > 0.2 | A, B, C, D | 3 | 1 | 3/3 = 1.00 | 1/2 = 0.50 |
| > 0.0 | all | 3 | 2 | 3/3 = 1.00 | 2/2 = 1.00 |

**Step 2 — plot each (FPR, TPR) pair**

You get the points: (0,0) → (0, 0.33) → (0, 0.67) → (0.5, 0.67) → (0.5, 1.0) → (1.0, 1.0)

**Step 3 — compute AUC**

Area under that staircase curve. In this case ≈ **0.83** — the model is fairly good but not perfect (D was ranked below C, a negative).

That one misordering is exactly what AUC penalizes — it is directly counting the fraction of (positive, negative) pairs ranked correctly.

# Threshold tuning

Как подбирать порог для классификации?

- Threshold по умолчанию (0,5) часто отдает предпочтение мажоритарному классу в imbalanced dataset, что приводит к низкой полноте/точности для миноритарного класса (positives, `1`). [threshold-moving-for-imbalanced-classification](https://machinelearningmastery.com/tour-of-evaluation-metrics-for-imbalanced-classification/).
- Пример: Модель может достичь 99% точности, всегда предсказывая доминирующий в выборке класс, при этом проигнорировав все критически важные случаи миноритарного класса (например, мошенничество или заболевание).
